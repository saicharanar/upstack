Fetching data in a Server Component is refreshingly plain: the component is **`async`**, and you
**`await`** your data right in the body. No `useEffect`, no `isLoading` state, no client round-trip
‚Äî the HTML arrives with the data already in it.

## Await data in the component

```jsx
async function ProductPage({ params }) {
  const { id } = await params;
  const product = await fetch(`https://api.example.com/products/${id}`).then((r) => r.json());
  return <h1>{product.name}</h1>; // rendered on the server, with data
}
```

Compare that to the client pattern (`useEffect` + `useState` + loading/error) from the React
course ‚Äî on the server it collapses to a single `await`.

## Reading the query string with searchParams

Pages receive **`searchParams`** ‚Äî the `?q=‚Ä¶&sort=‚Ä¶` part of the URL ‚Äî so filters, search, and
pagination live in the URL and drive what you fetch:

```jsx
async function Search({ searchParams }) {
  const { q, sort } = await searchParams;
  const results = await search(q, sort);
  return <Results items={results} />;
}
```

<Callout type="tip" title="Keep filters in the URL">
  Putting search/sort/page in `searchParams` makes state **shareable and bookmarkable**, and lets
  the server render the right results directly. Normalizing those raw params into a clean query is
  plain logic ‚Äî which is your exercise.
</Callout>

## Recap

- **Server Components can be `async`** and **`await`** data directly ‚Äî no effect, no loading state.
- Pages read the query string via **`searchParams`** to drive what they fetch.
- Keep filters in the URL: shareable, bookmarkable, and server-renderable.

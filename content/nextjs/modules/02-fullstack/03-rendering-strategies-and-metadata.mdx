The old Next had separate functions for static vs server rendering. The App Router unifies them:
you get **static**, **dynamic**, or **incrementally revalidated** pages depending on how a route
fetches data — the caching model from earlier, seen from the rendering side.

## Three strategies, one router

- **Static (SSG)** — rendered at **build time**, served as static HTML. The default when a route
  has no dynamic data. Fastest and cheapest; perfect for marketing pages and docs.
- **Dynamic (SSR)** — rendered **per request**. You get this when a route reads request-specific
  data (`cookies()`, `headers()`, uncached fetches, `force-dynamic`). Needed for personalized pages.
- **Incremental (ISR)** — static, but **refreshed on a timer** via `revalidate`. Static speed with
  data that stays reasonably fresh — great for a blog or catalog.

## Metadata for SEO

Export a **`metadata`** object (static) or a **`generateMetadata`** function (dynamic, per params)
and Next renders the right `<title>`, description, and Open Graph tags:

```jsx
export const metadata = { title: 'Blog — Acme', description: 'Posts about React.' };

// or per dynamic route:
export async function generateMetadata({ params }) {
  const { slug } = await params;
  const post = await getPost(slug);
  return { title: post.title, description: post.excerpt };
}
```

<Callout type="tip" title="Static by default is the point">
  Because routes are static unless they need to be dynamic, you get great performance for free and
  only "pay" for SSR where a page truly needs per-request data. Choose the strategy by the data's
  freshness needs, not by reaching for a different API.
</Callout>

## Recap

- **SSG** (build-time), **SSR** (per-request), **ISR** (revalidated) — one router, chosen by how you fetch.
- Request-specific data (`cookies`, `no-store`) makes a route **dynamic**; otherwise it's **static**.
- Export **`metadata`** / **`generateMetadata`** for per-page **SEO** tags.

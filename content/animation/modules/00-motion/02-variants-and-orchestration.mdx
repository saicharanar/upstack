Passing `initial`/`animate` objects inline is fine for one element. But when several elements
share states, or a parent needs to coordinate its children, you use **variants** — named sets of
values you refer to by string.

## Naming your states

Define an object of named states, then point `initial` and `animate` at the names:

```jsx
import { motion } from 'motion/react';

const box = {
  hidden: { opacity: 0, scale: 0.8 },
  visible: { opacity: 1, scale: 1 },
};

function Card() {
  return <motion.div variants={box} initial="hidden" animate="visible" />;
}
```

## Variants propagate to children

Here's the payoff: set variants on a **parent**, and its `initial`/`animate` labels flow down to
any child that has variants with the **same names**. You drive the whole group from the top:

```jsx
<motion.ul variants={list} initial="hidden" animate="visible">
  <motion.li variants={item} />   {/* inherits 'hidden' → 'visible' from the parent */}
  <motion.li variants={item} />
</motion.ul>
```

The children don't need their own `initial`/`animate` — the parent's labels reach them. This is
what makes coordinated, staggered entrances (next chapter) so tidy.

<Callout type="tip" title="Why variants beat inline objects">
  Named states keep animation logic readable, let one label drive many elements, and unlock
  orchestration (delays, stagger) between a parent and its children — impossible with inline
  `animate` objects alone.
</Callout>

## Recap

- **Variants** are named animation states (`hidden`, `visible`) referenced by string.
- `initial` and `animate` select a variant by **name**.
- Variants on a **parent propagate** to children sharing the same variant names — the basis of orchestration.

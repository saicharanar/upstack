Almost every real screen shows data that comes from somewhere else — an API, a database. The
mistake beginners make is thinking about only the happy path ("show the list"). Fetching actually
has **four** outcomes you must render, and forgetting any one is a real, visible bug.

## The four states of fetched data

## Fetching in an Effect

The classic way to load data on mount is a `useEffect` that fetches and stores the result in
state. Track the **status** so you can render the right branch, and clean up so a result that
arrives after the component unmounts is ignored:

```jsx
import { useState, useEffect } from 'react';

function Dashboard() {
  const [state, setState] = useState({ status: 'loading', users: [], error: '' });

  useEffect(() => {
    let active = true;
    fetchUsers()
      .then((users) => active && setState({ status: 'success', users, error: '' }))
      .catch((err) => active && setState({ status: 'error', users: [], error: err.message }));
    return () => { active = false; }; // ignore a late result after unmount
  }, []);

  if (state.status === 'loading') return <p>Loading…</p>;
  if (state.status === 'error') return <p>{state.error}</p>;
  if (state.users.length === 0) return <p>No users found.</p>;
  return <ul>{state.users.map((u) => <li key={u.id}>{u.name}</li>)}</ul>;
}
```

<Callout type="tip" title="In real apps, use a data library">
  Hand-written fetch-in-`useEffect` is fine to understand the states, but production apps usually
  use **TanStack Query (React Query)** or a framework's data layer. They hand you the same
  `{ isLoading, error, data }` shape — plus caching, retries, and refetching — so you still render
  exactly these four states.
</Callout>

## Recap

- Fetched data has **four states**: loading, error, empty, and success — render all four.
- Load on mount with **`useEffect`**, track a **status**, and **clean up** to ignore late results.
- Derive the UI from the status; don't forget the **empty** case.
- Real apps use a data library, but it hands you the same states to render.
